VI Server 替代 COM 的备份方案与风险评估

记录日期：2026-05-17

本文档用于保存“如果未来需要以另一条 VI Server 接入路径替代当前本地 COM/ActiveX 方案”的调研结论、风险分级与备份实施建议，供后续架构讨论、客户环境适配和应急决策时查阅。

一、先澄清概念

当前仓库并不是“COM 方案”和“VI Server 方案”完全分离的两套实现。现有正式运行时本质上已经是在通过 Windows Script Host + LabVIEW COM/ActiveX 访问 LabVIEW 暴露出来的 VI Server 对象模型。

  • 当前 worker 直接调用了 GetVIReference、SaveInstrument、PrintVIToHTML 等 VI Server 风格接口。
  • 现有架构还围绕这条链路补上了版本解析、目标实例匹配、持久 session host、全局串行化和缓存复用。
  • 因此，若只是把“COM”改称“VI Server”，这不是技术替换。
本次调研真正评估的对象：

如果未来要从“本地 COM/ActiveX 暴露的 VI Server 自动化”切到“另一条 VI Server 接入方式”，尤其是网络/端口型 VI Server，那么会带来哪些实施风险，以及如何把它作为备份方案而不是立即替换主路径。

二、当前主方案承担了哪些职责

当前方案不是单次脚本调用，而是一整层运行时能力组合。若未来替换，需要保证下面这些职责都能等价覆盖：

能力 当前实现承担的工作 替换时的影响
目标版本选择 根据目录标记、lvproj、VI 文件头解析所需 LabVIEW 版本，再从本机安装列表中选最匹配实例。 新方案必须继续保证“连接正确版本”，否则会出现错误实例写回。
实例定位与复用 可按 targetExe、安装目录、期望版本复用已有 LabVIEW 实例，避免误连。 若改网络 VI Server，需要重新定义如何锁定目标实例与连接地址。
全局串行化 通过自动化门卫保证任意时刻只发出一个自动化请求，降低 LabVIEW 冲突和崩溃风险。 新方案同样需要并发控制，不能假设网络接口天然安全。
功能覆盖 当前已支持属性读取、属性写回并保存、预览导出。 必须重新验证三条链路是否都能稳定工作，而不是只验证读取。
客户部署前提 当前只要求 Windows、本机安装 LabVIEW、WSH 与 COM 注册正常。 新方案若依赖网络 VI Server，通常要新增端口、防火墙、访问列表等配置要求。

三、风险结论

结论摘要

若未来只是更换本地封装层，但仍走本地 COM/ActiveX 暴露的 VI Server 对象模型，风险属于中等；若要改成网络/端口型 VI Server，则属于高风险运行时重构，不适合作为立即替换当前主路径的方案。

高风险点

  • 概念与边界风险：这不是局部 API 替换，而是运行时边界变更，会影响脚本宿主、实例定位、错误恢复与客户环境前提。
  • 版本绑定风险：当前实现可以精确选择本机匹配的 LabVIEW 安装；网络 VI Server 需要重新定义“连哪一台、哪一个版本、哪一个实例”。
  • 会话管理风险：现有 session host 支持复用与重建；网络连接还要额外处理断线、重连、端口不可达和会话漂移。
  • 功能等价风险：读取属性相对容易，但保存和预览导出依赖更强的本地实例语义，不能默认等价。
  • 运维与客户环境风险：网络 VI Server 往往意味着额外的防火墙、端口、白名单和安全审计要求，交付成本明显升高。
  • 安全面扩张风险：当前主要是本机自动化；切到网络接口后，控制面从本地进程扩展到网络暴露面。

四、风险分级

替换目标 风险等级 说明
仅重封装本地 VI Server 调用方式 中 可能仍能复用现有版本解析、实例匹配和串行化模型，但要做完整回归验证。
改成网络/端口型 VI Server 高 需要重做连接模型、环境前提、故障恢复和安全策略，已经接近“新后端”而非“替换 COM”。
直接替换当前主路径 高 会同时触及属性读写、预览导出、会话复用与 UI 降级逻辑，不建议直接切换。

五、备份方案建议

更合理的做法不是“立刻用 VI Server 方案替换 COM 方案”，而是把它设计成一个受控的备份后端，只在明确的触发条件下试点。

建议的总体策略

  1. 继续保留当前 TypeScript + WSH + LabVIEW COM/ActiveX 主路径，不动默认行为。
  2. 把候选 VI Server 新方案做成实验后端，仅挂在 runtime 内部，不修改编辑器层公开接口。
  3. 通过显式配置或实验开关启用，禁止在未验证环境中自动切换。
  4. 任何时候只要新后端出现版本不确定、连接失败、导出不稳定等问题，都立即回退到当前主路径。

建议的接入边界

若要实现备份方案，优先保持以下公开能力签名不变，只替换内部后端：

  • readViProps
  • writeViProps
  • exportViPanelImages
  • hasReusableLabVIEWConnection

这样可以把影响限制在 runtime 层，避免同时改动编辑器会话、缓存协议和 WebView 消息流。

六、触发备份方案的条件

只有在下面场景成立时，才建议继续推进替代方案验证：

  • 客户环境明确限制当前 WSH/COM 自动化链路，且短期无法放行。
  • 存在跨机器控制 LabVIEW 的真实需求，本地 COM 已无法满足。
  • 可接受为客户环境新增网络端口、防火墙与 VI Server 安全配置。
  • 项目可承受一轮完整的功能等价验证，而不是只做读取能力验证。

七、最小可行验证范围

如果未来要真正评估备份方案，建议只做最小 spike，不要直接改主实现：

  1. 验证是否能稳定连接指定目标版本的 LabVIEW 实例。
  2. 验证 GetVIReference 等价能力。
  3. 验证属性写回后能否安全保存。
  4. 验证 PrintVIToHTML 或等价预览导出能力是否稳定。
  5. 验证断线、超时、版本不匹配时能否明确失败并回退，而不是误写入。

八、最终建议

当前项目最稳妥的决策是：

  • 把“网络/端口型 VI Server 替代本地 COM”视为备份研究方向，而不是当前交付主线。
  • 继续维护现有本地 COM/ActiveX 主方案，因为它已经和版本解析、实例匹配、会话复用、串行化和缓存逻辑深度耦合。
  • 若未来必须推进，只能以“实验后端 + 功能开关 + 明确回退路径”的形式逐步验证，不能一次性切主。
